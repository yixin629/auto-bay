## 4. 模块化单体架构

AutoBay 采用**模块化单体**（Modular Monolith）架构：

- 每个业务领域是一个独立模块（`modules/products/`、`modules/orders/` 等）
- 模块内部有清晰的四层结构：`models.py` → `schemas.py` → `service.py` → `router.py`
- 模块之间通过 UUID 外键和导入解耦
- 未来可以按需拆分为微服务（每个模块变成独立服务）

### 为什么不用微服务？

1. 当前团队规模小（1-3人），微服务增加运维复杂度
2. 模块边界已经清晰，拆分成本低
3. 单体部署简单、调试方便、事务一致性容易保证
4. 当某个模块成为瓶颈时（如图片处理），再单独拆出

## 5. 核心设计模式

### 5.1 MarketplaceConnector Protocol（平台集成抽象）

这是整个系统最关键的架构边界。所有业务逻辑通过统一接口调用平台 API：

```python
class MarketplaceConnector(Protocol):
    async def create_listing(self, data: ListingCreateDTO) -> ExternalListingResult
    async def update_stock(self, external_id: str, quantity: int) -> bool
    async def update_price(self, external_id: str, price: Decimal, currency: str) -> bool
    async def fetch_orders(self, since: datetime) -> list[ExternalOrderData]
    async def ship_order(self, external_order_id: str, tracking: TrackingInfo) -> bool
    async def fetch_messages(self, since: datetime) -> list[ExternalMessageData]
    async def send_message(self, thread_id: str, body: str) -> bool
    async def refresh_credentials(self) -> dict
    async def validate_connection(self) -> bool
```

新增平台只需实现这个接口 + 在 Registry 注册，业务代码零修改。

### 5.2 Product vs Listing 分离

- **Product** = 内部唯一真相（你自己的产品目录）
- **Listing** = 平台投影（同一产品在 eBay、Amazon、Shopify 可以有不同标题、价格、描述）

```
Product (SKU: ABC-001)
├── Listing: eBay AU  → AUD 49.99, title: "Premium Phone Case..."
├── Listing: Amazon US → USD 34.99, title: "Phone Case Compatible..."
├── Listing: Shopify  → AUD 45.00, title: "Our Best Phone Case"
└── Listing: TikTok   → AUD 39.99, title: "Trending Phone Case 🔥"
```

### 5.3 AI 分层模型路由

```
┌────────────────┐  ┌─────────────────────────┐  ┌───────────────────────┐
│  BUDGET Tier   │  │  STANDARD Tier          │  │  PREMIUM Tier         │
│  Claude Haiku  │  │  Claude Sonnet          │  │  Claude Opus          │
│                │  │                         │  │                       │
│ • 标题生成     │  │ • 描述生成               │  │ • 趋势分析综合         │
│ • 意图分类     │  │ • 客服回复               │  │ • 产品机会评估         │
│ • SEO 关键词   │  │ • 广告文案               │  │                       │
│ • 情感分析     │  │ • 产品翻译               │  │                       │
│               │  │                         │  │                       │
│ ~$0.001/1K tok│  │ ~$0.003/1K tok          │  │ ~$0.015/1K tok        │
└────────────────┘  └─────────────────────────┘  └───────────────────────┘
```

**核心原则**：能用规则的绝不用 LLM。定价/库存/汇率/利润评分全部用确定性算法。

### 5.4 重试 + 熔断

所有外部 API 调用都经过 resilience 层：

- **重试**：指数退避，最多 3 次，针对超时和 5xx 错误
- **熔断**：连续 5 次失败后断路 60 秒，避免雪崩
- **速率限制**：按平台配置调用频率

## 7. 数据流示例

### 产品上架到 eBay 的完整流程

```
1. 用户创建 Product (POST /api/v1/products/)
2. 用户点击 "AI Generate" → 调用 listing_generator agent
   → BUDGET 模型生成标题
   → STANDARD 模型生成描述
   → 保存到 product_translations
3. 用户创建 Listing (POST /api/v1/listings/)
   → 关联 product_id + platform_connection_id
4. 系统调用 ConnectorRegistry.get_connector("ebay", credentials, "AU")
   → 返回 EbayConnector 实例
5. EbayConnector.create_listing(data)
   → PUT /sell/inventory/v1/inventory_item/{sku}  (创建库存项)
   → POST /sell/inventory/v1/offer               (创建报价)
   → POST /sell/inventory/v1/offer/{id}/publish   (发布)
6. 保存 external_listing_id 到 Listing 表
7. Celery Beat 每 15 分钟执行 sync_inventory
   → 查询 inventory_items → 计算可用量 → 推送到 eBay
```

### 订单同步流程

```
1. Celery Beat 每 5 分钟触发 sync_all_orders
2. 遍历所有 active PlatformConnections
3. 对每个 connection:
   → connector.fetch_orders(since=last_synced_at)
   → 对比 external_order_id 去重
   → 新订单插入 orders + order_line_items
   → 更新 last_synced_at
```
